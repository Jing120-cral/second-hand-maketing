## 1. 项目背景

### 1.1 问题描述

在高校校园内，学生之间存在大量二手商品交易需求，例如：
- 毕业季离校时，大量教材、生活用品、电子产品需要转手；
- 学期初新生入学后，对低价二手物品有较强需求；
- 闲置物品临时转让（如键盘、台灯、自行车等）。

目前这些交易主要依赖 QQ 群、微信群、贴吧等非专用渠道，存在以下痛点：
- **信息零散**：商品信息散布在多个群聊中，难以检索和筛选；
- **信任缺失**：买卖双方互不相识，缺乏信用背书和交易保障；
- **管理困难**：缺少统一的商品审核、违规举报等管理手段；
- **效率低下**：买家难以快速找到需要的商品，卖家难以精准触达目标买家。

### 1.2 项目缘起

为解决上述问题，拟开发一套**河北民族师范学院二手交易平台**，仅面向本校在校生开放，通过学号实名登录，确保交易双方身份真实可靠，打造一个安全、高效、便捷的校内二手交易生态。

---

## 2. 项目目标

### 2.1 总体目标

建设一个面向本校学生的、以学号认证为基础的二手商品信息发布与交易撮合平台，成为校内唯一的官方二手交易渠道。

### 2.2 核心目标

| 目标 | 说明 |
|------|------|
| **实名可信** | 学号登录，一人一号，交易可追溯 |
| **信息聚合** | 集中展示所有在售商品，支持分类、搜索、筛选 |
| **流程规范** | 商品上架→审核→展示→交易→评价，全流程闭环管理 |
| **管理可控** | 管理员可审核商品、处理举报、管理用户 |
| **操作简洁** | 移动端优先，操作流程简单直观 |

## 3. 用户角色分析

### 3.1 角色定义

| 角色 | 说明 | 权限概要 |
|------|------|----------|
| **普通用户（学生）** | 所有在校生，通过学号登录 | 浏览商品、发布商品、购买商品、管理个人商品、评价、举报 |
| **管理员** | 由在校学生担任（招募或选拔），负责日常运营管理 | 商品审核、用户管理、举报处理、数据查看、公告发布 |
| **超级管理员** | 系统最高权限者（由负责老师担任） | 管理员账号管理、系统配置、数据导出、日志查看 |

> **说明**：普通用户在交易中既可以是买家也可以是卖家，无需切换身份。系统根据当前操作自动区分角色行为。

### 3.2 角色详细描述

#### 3.2.1 普通用户（学生）

- **买家行为**：浏览商品、搜索商品、按分类/价格筛选、查看商品详情、收藏商品、联系卖家、确认交易
- **卖家行为**：发布商品（含图片、描述、价格）、管理在售商品（编辑/下架/标记已售）、查看发布记录
- **个人行为**：管理个人信息（联系方式、收货地址等）、查看交易记录、查看收藏列表、评价交易对方、举报违规

#### 3.2.2 管理员（学生管理员）

- 由在校学生担任，经选拔或招募产生
- 审核用户发布的商品（通过/驳回）
- 查看和管理所有商品列表
- 处理用户举报（下架商品、警告用户、封禁账号）
- 查看平台数据统计
- 发布平台公告
- 接受超级管理员（老师）的监督和管理

#### 3.2.3 超级管理员

- 拥有管理员的所有权限
- 创建和删除管理员账号
- 查看系统操作日志
- 数据导出（交易数据、用户数据等）
- 系统参数配置

## 4. 功能性需求

### 4.1 用户认证模块

#### 4.1.1 学号登录

- **功能描述**：用户使用本校学号 + 密码进行登录。
- **学号验证方式**：采用**学号白名单**机制。由超级管理员预先将合法学号导入数据库（使用 Navicat 工具操作 MySQL），用户使用学号 + 密码直接登录。
- **当前阶段规则**：
  - **注册功能**：本期暂不开放自主注册。所有用户账号由管理员在后台预置创建。
  - **测试账号**：由管理员手动创建，可自定义学号和密码（无需遵循白名单格式限制，方便灵活测试）。
  - **正式账号**：管理员通过本地文档（Excel/CSV 等）获取合法学号列表，使用 Navicat 批量导入 MySQL 白名单表。
  - **学号格式**：本校学号为 14 位数字，示例格式：`2023030642006`
  - **默认密码**：正式预置账号的初始密码统一为 `0`（纯数字），用户首次登录后系统强制跳转到修改密码页面。
  - **后期规划**：后续版本将开放自主注册功能，届时注册时系统自动校验学号是否在白名单内
- **密码规则**：
  - 初始密码：`0`（登录后强制修改）
  - 自定义密码：不少于 6 位
  - 支持"忘记密码"功能（通过绑定的手机号或邮箱找回）
- **前置条件**：用户学号已由管理员预置在系统中
- **异常处理**：
  - 学号不存在（不在白名单中） → 提示"该学号未在系统允许范围内，如有疑问请联系管理员"
  - 密码错误 → 提示"密码错误，剩余尝试次数 X"
  - 连续错误 5 次 → 锁定账号 30 分钟

#### 4.1.2 个人信息管理

- 修改头像（支持上传）
- 修改昵称（默认显示为学号，可自定义）
- 绑定/更换手机号
- 绑定/更换邮箱
- 设置默认联系方式（微信/QQ/手机号，供买家查看）

### 4.2 商品发布模块

#### 4.2.1 发布商品

- **功能描述**：用户填写商品信息并提交上架。
- **信息字段**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 商品标题 | 文本 | 是 | 最多 60 字 |
| 商品描述 | 富文本 | 是 | 详细描述商品状况、使用时长、瑕疵等 |
| 商品图片 | 图片 | 是 | 至少 1 张，最多 9 张 |
| 商品分类 | 下拉选择 | 是 | 如：教材、电子产品、生活用品、服饰鞋包、运动器材、其他 |
| 商品成色 | 单选 | 是 | 全新 / 几乎全新 / 轻微使用痕迹 / 明显使用痕迹 |
| 原价 | 数字 | 否 | 购买时的价格（增强说服力） |
| 售价 | 数字 | 是 | 必须大于 0 |
| 交易方式 | 多选 | 是 | 面对面交易 / 校内自提 / 快递（校内） |
| 联系电话/微信 | 文本 | 否 | 选填，若不填则使用个人设置中的默认联系方式 |

- **发布流程**：填写表单 → 提交 → 进入"待审核"状态 → 管理员审核通过 → 上架展示
- **前置条件**：用户已登录，账号状态正常（未被封禁）
- **业务规则**：
  - 同一用户同时最多上架 20 件商品（可配置）
  - 每件商品通过审核后在架时间为 30 天，到期自动下架

#### 4.2.2 商品管理（卖家）

- 查看我发布的商品列表（按状态筛选：在售 / 已售出 / 待审核 / 审核驳回 / 已下架）
- 编辑商品信息（仅"待审核"和"在售"状态可编辑，进入审核状态的商品编辑后需重新审核）
- 下架商品（从在售列表移除）
- 标记为"已售出"（买家确认收货后自动标记，卖家也可手动标记）
- 查看商品浏览量、收藏量

### 4.3 商品浏览与搜索模块

#### 4.3.1 商品列表

- 首页默认按**热度排序**展示在售商品（瀑布流或列表形式）
  - 热度计算参考：浏览量 + 收藏量 + 近期交易活跃度
  - 热度排序目的在于让热门商品和优质商品优先曝光
- 支持按分类筛选
- 支持按成色筛选
- 支持按价格区间筛选
- 支持手动切换排序方式：
  - 按热度排序（默认）
  - 按发布时间排序（最新/最旧）
  - 按价格排序（从低到高/从高到低）
- 分页加载（每页 20 条）

#### 4.3.2 商品搜索

- 支持关键词搜索（匹配标题和描述）
- 搜索结果支持与商品列表相同的筛选和排序条件
- 支持保存搜索历史（本地）

#### 4.3.3 商品详情

- 展示商品所有信息（标题、图片轮播、描述、价格、成色、分类、发布时间、浏览量）
- 展示卖家信息（昵称、头像、历史在售数量、好评率）
- 提供"联系卖家"按钮（显示卖家联系方式）
- 提供"收藏"按钮
- 提供"举报"按钮

### 4.5 交易模块

#### 4.5.1 交易流程

> 注意：本系统为信息撮合平台，不涉及在线支付。实际交易由买卖双方线下完成。

```
买家联系卖家 → 双方沟通 → 线下交易 → 双方确认交易完成 → 互相评价
```

#### 4.5.2 交易状态管理

- **交易状态**：待确认 → 进行中 → 已完成 / 已取消
- **状态说明**：
  - 待确认：买家表示购买意向
  - 进行中：双方正在沟通或交易
  - 已完成：双方确认交易完成
  - 已取消：交易取消

#### 4.5.3 交易记录

- 买家可查看"我买到的"记录
- 卖家可查看"我卖出的"记录
- 每条交易记录包含：商品信息、交易时间、对方信息、评价状态

---

### 4.6 评价模块

- 交易完成后，买卖双方可互相评价
- 评价内容：好评 / 中评 / 差评 + 文字评价（选填）
- 评价影响卖家的好评率（展示在卖家信息中）
- 评价一经提交不可修改（但可追加评论）

### 4.7 举报模块

- 用户可举报违规商品（虚假信息、违禁品、欺诈等）
- 举报时需选择举报原因（预设选项 + 补充说明）
- 管理员在后台处理举报
- 举报处理结果通知举报人

### 4.10 NLP 自然语言处理功能

> **说明**：以下 NLP 功能为本平台的智能化增强模块，利用自然语言处理技术提升用户体验和平台管理效率。各功能标注优先级：🔴 核心 / 🟡 建议 / 🔵 可选。
> 
> **技术策略**：NLP/CV 功能统一采用**本地下载部署开源模型**的方式实现，零 API 调用成本。利用本机 **NVIDIA RTX 4060 + CUDA** 进行推理加速，同时保留 CPU 回退能力。模型选择策略：大模型（LLM、Embedding、OCR）走 GPU 推理，轻量模型（分词、规则匹配）CPU 即可。典型技术栈：
> - **CPU 运行**：jieba（分词）、fastText（分类）、SnowNLP（情感）、OpenCV（图像基础处理）
> - **GPU 加速**：Qwen2.5 / ChatGLM（本地大模型，CUDA 推理）、sentence-transformers（语义 Embedding，GPU 批量编码）、PaddleOCR（OCR 检测+识别，GPU 加速）、MobileNetV3 / EfficientNet（图像分类，ONNX Runtime GPU）
> - **硬件适配**：所有 GPU 模型均支持显存不足时自动回退 CPU，开发环境使用 `torch.cuda.is_available()` 自动检测设备

#### 4.10.1 违禁词/敏感词自动检测 🔴 核心

- **功能描述**：用户发布商品时，系统自动检测标题和描述中是否包含违禁词或敏感词，在提交前拦截或提交后标记高风险商品。
- **检测内容**：
  - 违法违规词汇（如违禁品名称、校园明令禁止交易的物品）
  - 不文明用语、辱骂性词汇
  - 诱导校外交易的词汇（如引导添加校外联系方式从事商业活动）
- **处理方式**：
  - 提交时实时检测 → 包含违禁词则**拒绝提交**，并红色高亮标注违规词汇
  - 疑似敏感词（不确定）→ 允许提交但标记为"高风险"，在管理员审核列表置顶提醒
- **技术实现**：
  - **基础层——本地词库**：管理员在后台**本地上传**违禁词库文件（Excel/CSV），导入 MySQL，使用 jieba 分词 + AC 自动机多模式匹配，毫秒级快速拦截常规违禁词。支持精确匹配 + 模糊匹配（同音字、拼音替代、特殊符号分隔等变体）
  - **增强层——本地大模型识别**：对于词库无法覆盖的变体表达和隐晦违规内容，调用**本地部署的大模型**（推荐 Qwen2.5-1.5B 或 ChatGLM3-6B-int4 量化版）进行语义级别内容审核，**GPU CUDA 推理**，单次检测 < 200ms。开发时自动检测 CUDA 可用性，无 GPU 时回退 CPU
  - 两层结合：本地词库保证毫秒级快速拦截，本地大模型兜底复杂隐蔽内容，全程无需联网
  - 大模型识别为疑似违规 → 标记"高风险"进入管理员人工复核
- **管理功能**：管理员可在后台维护违禁词库（本地上传更新、新增/删除/编辑词汇），大模型识别的误报/漏报可反馈给词库进行迭代优化

#### 4.10.2 商品分类智能推荐 🔴 核心

- **功能描述**：用户输入商品标题后，系统自动分析文本并推荐最可能的商品分类，减少用户手动选择的负担。
- **工作流程**：
  1. 用户在标题输入框中输入商品名称
  2. 系统实时分析标题关键词
  3. 在分类下拉框旁显示推荐分类（如："推荐分类：📚 教材"）
  4. 用户可点击一键选中，也可手动修改
- **示例**：
  - "高等数学第七版" → 推荐分类：教材
  - "iPhone 14 128G" → 推荐分类：电子产品
  - "九成新自行车" → 推荐分类：运动器材
- **技术实现**：
  - **基础层**：关键词规则 + 分类映射表，jieba 分词提取关键词，快速匹配常见商品
  - **增强层**：部署本地 **fastText** 轻量级文本分类模型（模型文件仅几 MB，毫秒级推理），覆盖规则无法匹配的新奇商品描述
  - 用户最终选择的分类可作为反馈数据，定期重新训练优化模型

#### 4.10.3 智能搜索增强 🟡 建议

- **功能描述**：在关键词搜索基础上增加语义理解能力，提升搜索结果的相关性。
- **增强能力**：
  - **同义词扩展**：搜索"电脑"也能匹配"笔记本"、"计算机"
  - **纠错建议**：输入"高等数雪" → 提示"您是否要搜索：高等数学"
  - **意图识别**：搜索"便宜的台灯" → 自动按价格从低到高排序
- **技术实现**：
  - 维护本地同义词词典
  - 语义相似度匹配：使用本地 **sentence-transformers**（如 MiniLM-L6 或 BGE-small-zh），**GPU 批量编码**商品标题为向量存入 MySQL，搜索时实时 Top-K 余弦相似度检索
  - 搜索纠错基于编辑距离 + 常见错词映射表

#### 4.10.4 评价情感分析 🔵 可选

- **功能描述**：对用户评价文本进行情感极性分析，自动识别差评并提醒管理员关注。
- **处理逻辑**：
  - 分析评价内容的情感倾向：正面 / 中性 / 负面
  - 自动识别差评并标记，在管理员后台的差评预警列表中展示
  - 当某卖家连续收到多差评时自动预警
- **技术实现**：
  - 使用本地 **SnowNLP**（中文情感分析库）进行情感极性判断，纯 Python 实现，无需外部依赖
  - 返回结果：正面 / 中性 / 负面 + 情感得分（0~1）

#### 4.10.5 商品描述质量评分 🔵 可选

- **功能描述**：用户填写商品描述后，系统给出描述质量评分和改善建议，引导用户写出更详细的描述。
- **评分维度**：
  - 文字长度（是否过于简短）
  - 信息完整性（是否包含使用时长、瑕疵说明等关键要素）
  - 可读性（段落是否清晰，是否有标点符号）
- **技术实现**：基于规则引擎（文字长度、标点、关键词覆盖度等维度计分）+ 本地部署轻量模型评估信息完整性和可读性，给出改善建议
- **展示方式**：描述输入框下方显示质量进度条和提示语，如"描述详细度 ⭐⭐⭐ 已达标"或"建议补充：商品的使用时长"

### 4.11 CV 计算机视觉功能

> **说明**：以下 CV 功能利用图像处理技术对商品图片进行分析和审核。各功能标注优先级：🔴 核心 / 🟡 建议 / 🔵 可选。

#### 4.11.1 图片质量检测 🔴 核心

- **功能描述**：用户上传商品图片时，系统自动检测图片质量，拒绝质量过低的图片，确保商品展示效果。
- **检测项目**：
  - **模糊检测**：使用拉普拉斯方差算法（Laplacian Variance）评估图片清晰度，模糊图片拒绝上传
  - **亮度过低**：检测图片平均亮度，过暗图片提示用户重新拍摄
  - **尺寸检测**：图片分辨率过低（如 < 200×200）时拒绝上传
  - **文件大小**：限制单张图片最大 5MB
- **用户提示**：
  - 模糊 → "该图片不够清晰，请重新拍摄"
  - 过暗 → "该图片光线较暗，建议在光线充足处拍摄"
  - 尺寸过小 → "图片尺寸过小，请上传更清晰的图片"

#### 4.11.2 图片内容审核 🟡 建议

- **功能描述**：系统自动检测上传的图片是否包含不当内容，辅助管理员审核。
- **检测内容**：
  - 是否包含人物面部（保护隐私，非必要情况下提醒用户）
  - 是否与商品无关（如表情包、纯文字截图、风景照等）
  - 可选升级：是否包含色情或暴力内容
- **处理方式**：
  - 检测到疑似不当内容 → 标记为"图片待人工审核"，仍允许提交但在管理员后台高亮提醒
  - 严重违规 → 直接拒绝提交
- **技术实现**：
  - **人物检测**：**OpenCV** Haar Cascade 人脸检测，轻量级，CPU 实时运行
  - **内容分类**：**MobileNetV3** 预训练模型（ONNX 格式，~20MB），本地推理判断是否与商品相关
  - **违规内容检测**：可选集成 **NSFW.js** 或本地轻量分类模型，检测不当图片
  - 审核结果返回违规类型 + 置信度，系统根据阈值决定拒绝/标记/放行

#### 4.11.3 商品图像分类识别 🟡 建议

- **功能描述**：根据用户上传的商品图片，自动识别商品类型，与 NLP 分类推荐互补验证。
- **工作流程**：
  1. 用户上传商品图片
  2. 系统使用图像分类模型预测商品类别
  3. 自动填充分类字段（用户可手动修改）
- **识别类别**：与商品分类对应（教材、电子产品、生活用品、服饰鞋包、运动器材等）
- **技术实现**：
  - 使用 **EfficientNet-B0** 或 **MobileNetV3-Small**（ONNX 格式，~15MB），在公开商品图像数据集上微调后本地部署
  - **ONNX Runtime GPU** 推理 < 30ms（vs CPU ~100ms），批量处理时优势明显
- **说明**：建议与 §4.10.2 商品分类智能推荐配合使用，NLP 从文本角度推荐，CV 从图像角度推荐，两者融合提高准确率。

#### 4.11.4 重复图片/相似图片检测 🔵 可选

- **功能描述**：检测新上传的商品图片是否与平台上已有的图片高度相似，防止重复发布或盗图。
- **检测场景**：
  - 同一卖家重复发布相同商品
  - 卖家盗用其他卖家的商品图片
- **技术实现**：
  - 计算图片感知哈希（pHash / dHash），比较汉明距离
  - 相似度超过阈值（如 90%）时提醒用户或标记待审核
- **处理方式**：检测到相似图片 → 提示"检测到该图片与已有商品相似，请确认是否为重复发布"

#### 4.11.5 图片文字提取（OCR）🔵 可选

- **功能描述**：对商品图片中的文字进行提取识别，辅助商品信息自动填充或违禁词检测。
- **应用场景**：
  - 教材图片 → 自动提取书名、作者等文字信息填充描述
  - 检测图片中的文字是否包含违禁词（如：图片中的联系方式水印）
- **技术实现**：
  - 使用 **PaddleOCR**（飞桨 OCR 引擎）本地部署，中文识别准确率高，支持倾斜/模糊/手写等复杂场景
  - 模型文件约 20MB，**GPU 加速**下检测+识别 < 100ms（CPU 约 300ms），开发时自动检测 CUDA 设备
  - 提取的文字信息可用于 NLP 分类推荐和违禁词检测

### 4.13 日志自动记录模块 🔴 核心

> **说明**：系统自动记录用户和管理员的所有关键操作，形成完整的操作审计链路。日志对用户不可见，仅超级管理员可查看。

#### 4.13.1 设计原则

- **全自动记录**：后端统一拦截，业务代码无需手动埋点。通过 Django 中间件（Middleware）或信号（Signal）机制实现。
- **区分角色**：日志明确标记操作来源是**普通用户**还是**管理员**（含超级管理员），两类日志在后台分开查看。
- **不可篡改**：日志一经生成不可修改或删除，保证审计完整性。
- **定期清理**：日志保留 90 天，超期自动清理（可配置）。

#### 4.13.2 用户操作日志

记录普通用户的所有关键行为。

| 操作类型 | 触发时机 | 记录内容 |
|----------|----------|----------|
| 登录 | 用户登录成功 / 失败 | 学号、IP 地址、登录时间、成功/失败、失败原因 |
| 修改密码 | 用户修改密码 | 学号、修改时间、IP 地址 |
| 修改个人信息 | 修改昵称/头像/联系方式等 | 学号、修改字段、旧值→新值、时间 |
| 发布商品 | 提交商品上架 | 商品ID、标题、分类、时间 |
| 编辑商品 | 修改商品信息 | 商品ID、修改字段、时间 |
| 下架商品 | 主动下架 | 商品ID、时间 |
| 标记已售 | 标记商品为已售出 | 商品ID、时间 |
| 收藏商品 | 收藏/取消收藏 | 商品ID、操作（收藏/取消）、时间 |
| 联系卖家 | 查看卖家联系方式 | 商品ID、卖家学号、时间 |
| 确认交易 | 确认交易完成 | 交易ID、商品ID、对方学号、时间 |
| 评价 | 提交交易评价 | 交易ID、评价等级（好评/中评/差评）、时间 |
| 举报 | 举报商品 | 商品ID、举报原因、时间 |
| 退出登录 | 主动退出 | 学号、时间 |

#### 4.13.3 管理员操作日志

记录管理员（含超级管理员）的所有管理行为。

| 操作类型 | 触发时机 | 记录内容 |
|----------|----------|----------|
| 登录/退出 | 管理端登录/退出 | 管理员账号、IP、时间 |
| 审核商品 | 通过/驳回商品 | 商品ID、操作结果、驳回原因、时间 |
| 强制下架 | 强制下架商品 | 商品ID、下架原因、时间 |
| 处理举报 | 处理用户举报 | 举报ID、处理方式（忽略/下架/警告/封禁）、处理意见、时间 |
| 封禁用户 | 封禁账号 | 目标学号、封禁原因、封禁时长、时间 |
| 解封用户 | 解封账号 | 目标学号、时间 |
| 发布公告 | 发布/编辑/删除公告 | 公告ID、操作类型、标题、时间 |
| 维护违禁词库 | 新增/编辑/删除违禁词 | 词汇、操作类型、时间 |
| 导入白名单 | 批量导入学号 | 导入文件名、导入数量、时间 |
| 创建管理员 | 创建/删除管理员账号 | 目标账号、操作类型、时间 |
| 修改系统配置 | 修改系统参数 | 配置项、旧值→新值、时间 |

#### 4.13.4 日志查看（超级管理员专属）

- **功能入口**：超级管理员后台 → 操作日志
- **筛选条件**：
  - 按角色筛选：用户操作 / 管理员操作 / 全部
  - 按操作类型筛选（下拉多选）
  - 按操作者搜索（学号或管理员账号）
  - 按时间区间筛选
  - 按 IP 地址搜索
- **展示字段**：时间、操作者、角色标识、操作类型、操作描述、IP 地址
- **导出功能**：支持将筛选结果导出为 Excel/CSV 文件
- **说明**：普通管理员**不可**查看操作日志，此功能仅对超级管理员（老师）开放

#### 4.13.5 技术实现要点

- **Django 中间件**：在请求链路中统一捕获用户 ID、IP、请求路径、请求参数，写入日志表
- **数据脱敏**：日志中的密码等敏感字段自动脱敏（如记录为 `***`）
- **异步写入**：日志写入采用异步方式（如 Django 信号 + 队列），不阻塞主业务流程
- **表分区**：日志表按月份分区，提高查询效率和清理效率

## 5. 非功能性需求

### 5.1 性能要求

| 指标 | 要求 |
|------|------|
| 页面加载时间 | 首页/列表页 < 2 秒 |
| 图片加载 | 支持缩略图和原图模式，缩略图 < 200KB |
| 并发用户 | 支持至少 500 人同时在线 |
| 接口响应时间 | 主要接口 < 500ms |

### 5.2 安全性要求

- 所有接口需进行身份验证（除登录/注册外）
- 密码使用加密存储（bcrypt 或类似方案）
- 全平台操作均自动记录日志，区分用户操作和管理员操作（详见 §4.13），日志不可篡改
- 防止常见 Web 攻击：SQL 注入、XSS、CSRF
- 图片上传需校验格式和大小，防止恶意文件上传
- 用户联系方式需做脱敏处理（仅登录用户可见）

### 5.3 可用性要求

- 系统可用性 ≥ 99.5%（非考试周）/ ≥ 99%（考试周低负载期）
- 维护时间应安排在凌晨，并提前通知用户

### 5.4 兼容性要求

- **平台定位**：Web 应用，采用响应式设计，同时适配桌面端和移动端浏览器
- **桌面端**：支持 Chrome、Edge、Firefox 最新两个版本
- **移动端**：支持 iOS Safari、Android Chrome 最新两个版本（通过响应式布局适配手机屏幕）
- **不涉及**：本期不开发微信小程序或独立 APP（后续可扩展）

### 5.5 数据要求

- 每日自动备份数据库
- 日志保留 90 天
- 商品图片保留至商品下架后 30 天

## 7. 业务流程设计

### 7.1 商品上架流程

```
[卖家] 填写商品信息 → [系统] 保存为"待审核" → [管理员] 审核
    ├── 通过 → [系统] 商品上架展示 → [卖家] 收到通知
    └── 驳回 → [卖家] 收到驳回原因 → [卖家] 修改后重新提交
```

### 7.2 交易流程

```
[买家] 浏览/搜索商品 → [买家] 查看商品详情 → [买家] 联系卖家
    → [线下] 双方沟通、看货、议价 → [线下] 完成交易
    → [任意一方] 在系统中确认交易完成
    → [双方] 互相评价
```

### 7.3 举报处理流程

```
[用户] 举报商品 → [管理员] 查看举报 → [管理员] 判断
    ├── 举报无效 → 忽略举报
    └── 举报有效 → 下架商品 / 警告卖家 / 封禁卖家
    → [举报人] 收到处理结果通知
```

| **商品 (Product)** | 发布的商品 | 标题、描述、分类、成色、原价、售价、交易方式、状态、浏览量、收藏量、发布时间 |
| **商品图片 (ProductImage)** | 商品多图（文件存本地，URL 存数据库） | 商品ID、图片URL、排序 |
| **分类 (Category)** | 商品分类 | 分类名称、图标、排序 |
| **收藏 (Favorite)** | 用户收藏 | 用户ID、商品ID、收藏时间 |
| **交易 (Transaction)** | 交易记录 | 商品ID、买家ID、卖家ID、状态、创建时间、完成时间 |
| **评价 (Review)** | 交易评价 | 交易ID、评价者ID、被评价者ID、评级、内容、时间 |
| **举报 (Report)** | 举报记录 | 商品ID、举报人ID、举报原因、状态、处理结果、处理时间 |
| **公告 (Announcement)** | 平台公告 | 标题、内容、发布时间、发布者 |
| **操作日志 (AuditLog)** | 全平台操作记录（用户+管理员） | 操作者ID、角色类型（用户/管理员/超管）、操作类型、操作对象ID、操作描述、IP地址、User-Agent、操作时间、请求耗时 |
| **学号白名单 (StudentWhitelist)** | 合法学号列表 | 学号、导入时间、导入者ID |
| **违禁词库 (ForbiddenWord)** | NLP 违禁词/敏感词 | 词汇、级别（违禁/敏感）、创建时间、添加者ID |
| **图片审核记录 (ImageAuditLog)** | CV 图片审核结果 | 商品ID、图片URL、检测类型、检测结果、置信度、时间 |

> **说明**：商品图片文件存储在本地服务器磁盘，数据库中（MySQL，通过 Navicat 管理）仅保存图片 URL 路径。学号白名单由超级管理员通过 Navicat 批量导入 MySQL。违禁词库由管理员在后台维护。

### 8.2 主要数据关系

```
User (1) ─────< (N) Product         （一个用户可发布多个商品）
Product (1) ──< (N) ProductImage    （一个商品可有多张图片）
Product (1) ──< (N) Favorite        （一个商品可被多人收藏）
Product (1) ──< (N) Transaction     （一个商品对应一条交易记录）
Transaction (1) ──< (N) Review      （一条交易可产生双向评价）
Product (1) ──< (N) Report          （一个商品可被多次举报）
```

## 10. 风险与约束

### 10.1 主要风险

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 用户身份冒用（借用学号） | 信任体系受损 | 登录增加手机验证码；敏感操作二次验证 |
| 线下交易纠纷 | 平台公信力下降 | 明确平台免责声明；提供举报和申诉通道 |
| 违禁品上架 | 法律/校规风险 | 商品关键词过滤；管理员审核机制；举报机制 |
| 高并发压力（毕业季） | 系统性能下降 | 做好缓存策略；图片懒加载；分页优化 |
| 用户量不足 | 平台冷启动困难 | 初期运营推广；与学生会/社团合作宣传 |

### 10.2 约束条件

- 用户群仅限本校在校生，通过学号白名单验证（由管理员使用 Navicat 导入 MySQL）
- 本期不开放自主注册，账号由管理员预置，默认密码为 `0`
- 平台不涉及在线支付，交易由买卖双方线下完成
- 平台形态为 Web 应用（响应式），本期不涉及小程序或 APP
- 不提供站内即时通讯，沟通由用户通过联系方式线下进行
- 当前阶段仅需本地部署，开发笔记本配置：AMD Ryzen 7 7735H + NVIDIA RTX 4060，大模型推理走 GPU CUDA 加速，硬件足够
- 管理员由学生担任，超级管理员由老师担任
